# Confirmed Proposal Guide

After `/make-proposals` has produced two candidate proposals, the user selects one and describes any adjustments. This command uses the **selected proposal + user adjustments** to produce **one** final, actionable proposal document.

- Use the SwitchMode tool to switch to **agent** mode.
- Write the final output to `docs/proposals/proposal-[feature].md` in the project (same path as make-proposals; if a dual-proposal document already exists, **overwrite** it with the confirmed single-proposal version).

## Prerequisites

The user should provide the following in this conversation (ask follow-up questions if anything is missing):

1. **Selected proposal:** `Proposal 1` / `Proposal 2`, or the proposal title (matching make-proposals output).
2. **Adjustments:** what to change, keep, or add relative to the selected proposal (preserve the user's wording and structure it clearly).
3. **Optional context:**
   - Path to an existing `docs/proposals/proposal-*.md` (make-proposals output);
   - Research document path (e.g. under `research/` or `docs/` from `/research-codebase`);
   - Original requirement or feature slug (for file naming).

If the user does not provide research or dual-proposal paths, search the repo for `docs/proposals/proposal-*` and relevant files under `research/`; if nothing can be linked, list the gaps and ask follow-up questions.

## Guidelines

- Treat `/make-proposals` and `/research-codebase` documentation conventions as **authoritative** (cite code as `` `path:line` ``).
- Output **only one** final proposal; do not present two parallel candidates.
- User adjustments have **highest priority**: when they conflict with the selected proposal, follow the user and document the trade-off.
- **Do not** produce an executable development plan (that is `/make-plan`'s job); only produce the confirmed proposal.
- Stay concise and prioritize clarity; key changes must trace back to research and source code.

## Steps

1. **Receive confirmation**
   - Record: feature slug, selected proposal number/title, user adjustment text, related document paths.
2. **Read upstream materials**
   - Read the dual-proposal document from make-proposals, if it exists.
   - Read the research document and extract constraints, modules, dependencies, data flows, and prior decisions.
3. **Synthesize the final proposal**
   - Use the selected proposal as the skeleton and merge user adjustments item by item.
   - Remove or de-emphasize content from the unselected proposal; optionally add 1–2 sentences on why the other proposal was not chosen (if the user explained it).
   - Update: key changes, affected paths, trade-offs, validation steps, open questions (including new gaps from adjustments).
4. **Name and write the file**
   - File name: `docs/proposals/proposal-[feature].md` (match existing proposal naming; English kebab-case is fine).
   - Write using the template below; in the reply, give the file path and a 3–5 bullet summary.
5. **Deliver**
   - Write the file to disk; do not paste the full document in chat only (a summary is fine).

## Output Template

Use this when writing `docs/proposals/proposal-[feature].md`:

---

## Confirmed Solution Proposal

**Status:** Confirmed (generated by `/create-proposals`)

**Context:**
- **Request:** `<brief restatement of the requirement>`
- **Research sources:** `<file names and key sections; or "research gap">`
- **Upstream proposal:** `<original make-proposals file path; or "user input only">`
- **Selected base:** `<Proposal 1/2 or title>`
- **User adjustment summary:** `<structured list of confirmed changes, preserving intent>`

### Current State Summary (optional)

`<Keep a concise version from research or dual-proposal docs if available; omit if not>`

---

### Final Proposal – `<title>`

- **Overview:** `<2–4 sentences including adjusted goal>`
- **Changes vs selected proposal:** `<bullet list of differences caused by user adjustments>`
- **Key changes:** `<components/modules/file paths>`
- **Trade-offs:** `<risks and benefits, including new risks from adjustments>`
- **Validation:** `<tests/experiments/acceptance steps>`
- **Open questions:** `<items still unclear or needing follow-up research>`

### Rejected Alternative (optional)

- **Not selected:** `<other proposal title>`
- **Reason:** `<user or analysis reason; omit if not stated and not obvious>`

---

## Notes

- When user adjustments conflict with the selected proposal, **follow the user** and document the difference under "Changes vs selected proposal".
- If research does not cover the request, list gaps under "Open questions" and recommend `/research-codebase`.
- Do not invoke `/make-plan` or write an implementation plan file; use `/make-plan` only after the proposal is confirmed.
- Write replies and file content in English.
