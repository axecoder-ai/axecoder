# 选型记录

## 一句话需求回顾

Software Co. 应与 Agent 共享同一 Agent 循环，效率对齐；SOP（PRD/设计/任务/QA）改为软约束与可选交付物，不再用硬状态机拆成多轮 LLM。

## 方案对比表

| 维度 | 提案 1 单 Session 软 SOP | 提案 2 保留硬编排局部合并 |
|------|---------------------------|---------------------------|
| 核心思路 | 默认单 session 连续跑，SOP 写进 prompt | 仍多 session，合并部分阶段 |
| 主要改动范围 | sop-fast-pipeline、agent-loop、workshop-ipc | sop-pipeline-engine、task-runner |
| 优点 | 与 Agent 同效率，根治 session 碎片化 | 改动小、风险低 |
| 缺点 / 风险 | 交付物结构化靠模型自觉 | 仍多轮 LLM，未达第一性原理 |
| 工作量 | 中 | 小 |
| 适合场景 | 日常开发 + 可追溯 SOP | 仅「快一点」的折中 |

## 关键差异

- 提案 1 拆掉回合边界；提案 2 保留。
- 提案 1 工具全开；提案 2 仍角色阉割。
- 提案 1 保留严格模式回退；提案 2 无架构升级。

## 推荐方案

**推荐：提案 1 – 单 Session 软 SOP**

与第一性原理一致，用户已明确要 Agent 级效率。

## 用户最终选择

- **选定：** 提案 1 – 单 Session 软 SOP（默认 Fast 路径）
- **调整说明：** 无额外调整
