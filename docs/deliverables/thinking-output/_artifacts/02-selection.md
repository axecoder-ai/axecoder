# 步骤 2：方案选型记录

## 2a. 选型摘要

### 一句话需求回顾
在 AI 回答问题时，将 thinking 过程中有内容的部分（Model Call 和 Tool Result）在前端 UI 中展示出来，与最终回答内容分离呈现。

### 方案对比表

| 维度 | 方案 A：分离 Delta 通道 | 方案 B：结构化增强 | 方案 C：完整重构 |
|------|-------------------------|-------------------|-----------------|
| 核心思路 | 修改 IPC payload，区分 content 和 reasoning 流 | 在 A 基础上解析 reasoning 为结构化步骤 | 独立 ThinkingPanel 组件 + token 统计 + 搜索 |
| 主要改动范围 | 4 个文件：agent-loop、IPC types、Vue 组件 | 6 个文件：A 的基础 + parser + 状态管理 | 10+ 文件：新组件、样式、状态、工具函数 |
| 优点 | 最小改动、立即见效、风险低、易回退 | 可读性好、可扩展、增量升级 | 最佳体验、完整功能、专业交互 |
| 缺点 / 风险 | 功能简单，只显示原始文本 | 解析逻辑可能不稳定 | 工作量大、改动面广、测试成本高 |
| 工作量（粗估） | 小（2-3h） | 中（5-6h） | 大（15-20h） |
| 适合场景 | 快速验证需求、MVP 阶段 | 需要一定可读性，但控制成本 | 追求最佳用户体验、长期产品化 |

### 关键差异说明

1. **展示形式**：
   - 方案 A：纯文本展示，简单折叠面板
   - 方案 B：结构化卡片，区分工具调用、结果、推理
   - 方案 C：独立面板，支持展开/折叠、搜索、复制、导出

2. **技术复杂度**：
   - 方案 A：仅需修改流式数据分发逻辑
   - 方案 B：需增加解析器和状态管理
   - 方案 C：需完整的组件体系和交互设计

3. **可维护性**：
   - 方案 A：代码简单，易维护
   - 方案 B：中等复杂度，解析逻辑需维护
   - 方案 C：组件化好，但代码量大

4. **扩展性**：
   - 方案 A：后续升级到 B/C 需重构
   - 方案 B：可平滑升级到 C
   - 方案 C：为未来需求（token 统计、导出等）预留空间

5. **用户体验**：
   - 方案 A：基本可用，体验一般
   - 方案 B：可读性明显提升
   - 方案 C：专业级交互，最佳体验

6. **风险控制**：
   - 方案 A：改动小，风险低，可快速回退
   - 方案 B：中等风险，解析错误可能影响显示
   - 方案 C：大改动，需完整测试，回退成本高

### 推荐方案

**推荐：方案 A – 分离 Delta 通道**

**推荐理由**：
1. **性价比最高**：2-3 小时即可解决核心需求（显示 thinking 内容）
2. **风险可控**：改动仅 4 个文件约 50 行代码，不破坏现有架构
3. **快速验证**：可立即上线使用，收集真实用户反馈
4. **渐进式升级**：完成后可根据反馈逐步升级到 B 或 C，不浪费投入

**何时应改选其他方案**：
- 如果团队对用户体验要求极高，且有充足开发时间（15-20h），选方案 C
- 如果需要平衡体验和成本，且能接受 5-6h 工作量，选方案 B

下一步将通过选择题确认最终方案。你可以选择推荐的方案 A，或根据项目实际情况选择 B/C。若有特殊调整需求（如"只做 token 统计不做搜索"），可在调整说明中注明。完整技术细节已保存在 `docs/proposals/proposal-thinking-output.md`。
