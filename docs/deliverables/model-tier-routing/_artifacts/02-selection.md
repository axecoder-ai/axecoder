# 选型记录：model-tier-routing

## 2a 选型摘要

### 一句话需求回顾

为 AxeCoder 增加「复杂用主模型、简单/探索用快速模型」的分流能力（类 同类 Agent 双模型），在现有多模型配置之上扩展，不改变用户手动选当前模型的基本习惯。

### 方案对比表

| 维度 | 提案 1 规则分流 | 提案 2 Agent 可选 model | 提案 3 仅子代理 |
|------|-----------------|-------------------------|-----------------|
| 核心思路 | 设置主+快，主进程按任务类型选模型 | 同 1 + LLM 传 model 参数 | 只改子代理路径 |
| 主要改动范围 | models 存储、resolve、agent/workshop | + 工具 schema、提示词 | agent-subagent、workshop explore |
| 优点 | 可预测、易测 | 灵活 | 改动最小 |
| 缺点 / 风险 | 规则粗 | 依赖 LLM 传参 | 主会话无法分流 |
| 工作量 | 中 | 中–大 | 小 |
| 适合场景 | 默认推荐 | 需细粒度子任务选模 | 快速试水 |

### 关键差异说明

- 提案 1/3 都不让 LLM 选模型；提案 2 增加工具参数。
- 提案 3 不做设置页「双槽位」完整 UX，仅一个 fast 字段。
- 第一期均不做 ML 复杂度分类，靠 `main` vs `subagent` 任务类型。

### 推荐方案

**推荐：提案 1 – 设置双模型 + 主进程规则分流**

**推荐理由：** 与现有 `activeModelId`、子 Agent、工坊 explore 路径契合；行为可单测；不增加 LLM 漏参风险。

### 选型提示

用户通过 AskQuestion 确认；若跳过则采用推荐方案。

## 2b 用户最终选择

- **选定提案：** 提案 1 – 设置双模型 + 主进程规则分流
- **调整说明：** 用户跳过 AskQuestion；默认 **保留 `activeModelId` 语义**（即主模型），新增 `fastModelId`；协作工坊经理等主发言仍用 `activeModelId`，仅 explore/子 Agent 走快速模型。
