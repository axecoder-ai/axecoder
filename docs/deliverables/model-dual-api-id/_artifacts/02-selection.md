# 选型记录：model-dual-api-id

## 2a 选型摘要

### 一句话需求回顾

在「添加/编辑模型」中为**同一配置**提供快速与深度两个 API 模型 ID，并在回答时根据问题复杂度**自动**选用；废弃全局「快速模型（子任务）」第二条目下拉。

### 方案对比表

| 维度 | 提案 1 启发式路由 | 提案 2 LLM 分类 | 提案 3 仅 UI |
|------|-------------------|-----------------|--------------|
| 核心思路 | 规则判断 fast/deep | 额外 LLM 分类 | 双 ID 仅配置，仍按任务类型 |
| 主要改动 | ModelEntry + heuristic + 调用链 | 同左 + 分类调用 | 表单与存储 |
| 优点 | 无额外成本、可单测 | 语义更准 | 最小 |
| 缺点 | 可能误判 | 延迟与成本 | **不满足自动判断** |
| 工作量 | 中 | 大 | 小 |
| 适合场景 | V1 落地 | 后续增强 | 不推荐 |

### 关键差异说明

- 提案 1/2 均满足「单条目双 API ID」；仅 1 在主会话按**问题**选档。
- 提案 2 每轮可能多一次 API。
- 提案 3 不做复杂度自动判断，不符合本轮目标。
- 与既有 `model-tier-routing` 区别：从「两个 ModelEntry」改为「一个 Entry 两个 API 名」。

### 推荐方案

**推荐：提案 1 – 单条目双 API ID + 启发式复杂度路由**

理由：零额外 API、与现有 agent 循环易集成；误判可通过规则迭代；LLM 分类可 V2。
