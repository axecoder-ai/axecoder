# 选型记录 — main-process-isolation

## 2a 选型摘要

**需求回顾：** 优化 AxeCoder 主进程承载过多 Agent 业务（57 个工具、loop、MCP、CodeGraph 等）导致的隔离性弱问题，向 VS Code「主进程薄桥接 + 子进程执行业务」靠拢，但不强求完整 Extension Host。

### 方案对比表

| 维度 | 提案 1 – 分层渐进：危险/重型工具子进程化 | 提案 2 – 独立 Agent Worker 进程 |
|------|------------------------------------------|----------------------------------|
| 核心思路 | loop 与审批留主进程；重型工具迁入 Tool Worker | 整个 Agent 运行时外移到独立进程 |
| 主要改动范围 | 新增 agent-worker + 改造 tool-executor 路由 | 搬迁 electron/main/agent/ + IPC 代理层 |
| 优点 | 改动可控；复用 LSP/terminal 模式 | 隔离最强；架构接近 VS Code EH |
| 缺点 / 风险 | loop 未隔离；RPC 与审批需设计 | 工作量大；交叉调用重设计 |
| 工作量（粗估） | 中 | 大 |
| 适合场景 | 分期演进、最小改动 | 一次性大重构、长期独立 runtime |

### 关键差异

- 提案 1 只拆工具执行；提案 2 整包外移。
- 提案 1 可 Phase1 只迁 Bash+Web；提案 2 需全量搬迁才闭环。
- 提案 1 与 browser-runner、LSP 模式一致；提案 2 需新建能力代理层。
- 两者都不是 VS Code 扩展市场。

### 推荐方案

**推荐：提案 1 – 分层渐进：危险/重型工具子进程化**

理由：符合现有子进程桥接实践，改动面可控，优先解决 Bash/浏览器/CodeGraph 阻塞主进程；与最小改动原则一致。

## 2b 用户最终选择

- **选定提案：** 提案 2 – 独立 Agent Worker 进程（整包 Agent Runtime 外移）
- **调整说明：** 无额外调整，按方案原文落地
