# 需求：Workshop 群聊重做

**状态：** 需求澄清完成（由 `/clarify` 生成）

**日期：** 2026-06-03

---

## 背景

现有 **Collab Workshop** 群聊功能不好用：实现过于复杂，且未能按预期跑通。用户希望**完全重做**，放弃当前的「技术经理先拆步骤 → 逐步执行 → 每步验收」模式，改为更贴近真实群聊的**轮流说话 + AI 调度话语权**机制。

---

## 目标

在 AxeCoder 内提供一种 **Workshop 类型**的协作会话：BOSS（当前用户）发起任务后，由 AI 从已配置的 Users 成员中调度接话人；成员以独立 Agent（可读代码、可执行工具）完成工作并发言；技术经理负责代替 BOSS 发言并派活；仅在需要澄清时才暂停等待 BOSS 回答。会话由技术经理判定任务完成后结束。

---

## 角色与身份

| 角色 | 系统标识 | 是否在 Users 列表 |
|------|----------|-------------------|
| **BOSS**（当前用户） | 设置页「我的资料」（`profileDisplayName` / 头像） | 否 |
| **技术经理** | Users 内置条目（`builtinRole: manager`） | 是 |
| **其他成员**（后端、前端、测试等） | Users 中用户添加的条目 | 是 |

- 成员档案（头像、姓名、角色、擅长领域）**复用现有 Users 配置**，不另建一套。
- 技术经理仍为内置、不可删、角色/擅长不可改（与现有 Users 规则一致）。

---

## 会话入口与 UI

- **与 Agent 共用同一 ChatPane**（中央聊天区域），通过会话**类型**区分：`Agent` / `Workshop`。
- 用户点击 **Workshop 按钮**进入/创建 Workshop 类型会话（不再维护独立、割裂的 Workshop 面板作为主入口）。
- 群聊消息应能展示各成员身份（头像、昵称、角色等），具体展示方式不在本需求范围限定。

---

## 核心对话流程

以下描述**业务行为**，不涉及具体技术实现。

### 1. 启动

用户通过 Workshop 按钮进入 Workshop 会话，**BOSS 发送第一条消息**（任务/需求描述），协作开始。

### 2. BOSS 发言后 → AI 选下一个接话人

BOSS 每发一条消息后，由 **AI 根据消息内容与 Users 成员列表**，判断**下一个应该说话的人**（必须是已配置的成员之一）。

### 3. 成员发言

被选中的成员以 **独立 Agent（具备工具能力）** 执行任务并输出发言内容；发言完成后进入下一环节。

### 4. 成员说完 → AI 判断话语权交给谁

成员发言结束后，由 **AI 判断**话语权应交给：

- **BOSS**：仅当需要向 BOSS **澄清或确认**时（对应 Agent 体系中的 `AskUserQuestion` 场景）；此时群聊**暂停**，等待 BOSS 在输入框回答。
- **技术经理**：其余情况交给技术经理。

> BOSS **不会**在每一轮都参与对话；只有需要澄清时才介入。

### 5. BOSS 澄清后

BOSS 通过 `AskUserQuestion` 回答完毕后，再次由 **AI 从 Users 成员中选出下一个接话人**（回到步骤 2 的逻辑）。

### 6. 技术经理接话

话语权交给技术经理时，技术经理应：

- **主要代替 BOSS 说话**（总结、指令、协调等）；
- **同时决定下一个任务应交给谁**（从 Users 成员中指定）。

技术经理点名后，**被指定的成员应立刻开始发言**，**不再**额外经过一轮「该谁接话」的 AI 路由。

### 7. 结束

当 **技术经理判断任务已完成** 时，宣布本轮协作结束。

---

## LLM 消息角色映射（API 层）

向大模型发送对话历史时，角色映射规则如下：

| 发言者 | API `role` |
|--------|------------|
| BOSS | `user` |
| 技术经理 | `user` |
| 其他所有成员 | `assistant` |

此规则用于保证模型侧对话结构的语义一致性；与 UI 上展示的多人头像/昵称相互独立。

---

## 功能要求

1. **完全取代**现有 Workshop 的拆步 / 逐步执行 / 经理验收（VERIFY / redo / abort）流程。
2. **复用 Users** 作为协作成员来源；BOSS 使用「我的资料」，不入 Users 列表。
3. **AI 路由接话人**：BOSS 发言后、以及 BOSS 澄清回答后，由 AI 选择下一个成员。
4. **AI 路由话语权**：成员发言后，由 AI 决定交给 BOSS（澄清）或技术经理。
5. **技术经理派活即执行**：经理指定成员后，该成员立即发言，无需二次路由。
6. **成员具备 Agent 能力**：除 BOSS / 技术经理外的成员发言时，应能读代码、调用工具（与现有子 Agent 能力一致）。
7. **BOSS 澄清机制**：仅在需要澄清时暂停；通过 `AskUserQuestion` 等待 BOSS 回答。
8. **会话类型**：Workshop 与 Agent 共用 ChatPane，以类型字段区分。
9. **入口**：Workshop 按钮创建/进入 Workshop 会话。
10. **结束条件**：由技术经理判定任务完成。

---

## 非功能要求

1. **实现应比现有 Workshop 更简单**，避免过度复杂的状态机与多层嵌套流程。
2. **可测试性**：核心调度逻辑（路由、话语权切换、角色映射）应可独立验证。
3. **与 Agent 数据隔离**：Workshop 会话与 Agent 会话的持久化、上下文应相互独立（延续现有隔离原则）。

---

## 明确不在本需求范围内

- 具体 UI 视觉 redesign 细节（在共用 ChatPane 前提下，样式可沿用或微调）。
- 自动写盘 / diff 确认（除非后续单独提出）。
- 真多人远程协作（Phase 2 能力）。
- 具体模型选型、每角色绑模策略（沿用现有配置即可，不另定规则）。
